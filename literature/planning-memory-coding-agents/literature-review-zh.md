# 文献综述：基于LLM的编程智能体中的规划与记忆（上下文工程）技术

**日期：** 2026-03-19
**范围：** 聚焦型（2023–2026）
**调研论文数：** 约45篇核心工作
**主题：** LLM智能体中规划与记忆/上下文工程技术的融合，重点关注编程智能体

---

## 1. 引言

基于LLM的编程智能体——能够自主浏览代码仓库、定位缺陷、生成补丁并验证修复的系统——已迅速成为大语言模型最具影响力的应用之一。SWE-agent（Yang et al., 2024）、OpenHands（Wang et al., 2024）、Devin（Cognition AI, 2024）和Claude Code（Anthropic, 2025）等系统证明，配备工具、规划和记忆能力的LLM能够处理真实的软件工程任务。

然而，两个根本性挑战制约着这些智能体：

1. **规划复杂性**：真实的编程任务需要多步推理——故障定位、理解跨文件依赖、制定修复策略、以及对失败尝试进行迭代。扁平的反应式智能体循环（如基础ReAct）在长周期、多文件任务上表现不佳。

2. **上下文限制**：即使拥有100万token的上下文窗口，智能体仍面临*上下文漂移*（偏离原始目标）、*认知过载*（工作记忆被无关观察信息充斥）和*信息丢失*（关键推理在上下文管理过程中被丢弃）等问题。

本综述调研了**规划**与**记忆/上下文工程**交叉领域的前沿研究，特别关注编程智能体。我们将文献组织为四个主题：（1）规划架构，（2）记忆系统，（3）上下文管理策略，（4）编程智能体集成系统。

## 2. LLM智能体的规划架构

### 2.1 反应式与线性规划

基础智能体范式是**ReAct**（Yao et al., 2023），它在线性链中交替执行推理轨迹与动作。虽然对简单任务有效，但ReAct缺乏回溯能力，在早期错误不断累积的长周期任务上性能退化。

**Reflexion**（Shinn et al., 2023）通过语言自我反思扩展了ReAct：在任务失败后，智能体批评其轨迹并将语言反馈存储在情景记忆缓冲区中，在未来尝试时作为前缀注入。这使得智能体能够从错误中学习，而无需更新模型权重。Reflexion在HumanEval（pass@1: 91.0%）和ALFWorld上取得了显著提升。

### 2.2 树结构与搜索式规划

**Tree of Thoughts (ToT)**（Yao et al., 2024）将思维链泛化为允许分支探索多条推理路径，并支持评估与回溯。这种结构化的推理方式对存在多种可行方案的任务尤为有价值。

**Language Agent Tree Search (LATS)**（Zhou et al., 2024; ICML 2024）通过将蒙特卡洛树搜索（MCTS）集成到智能体循环中，统一了推理、行动和规划。LLM同时充当智能体、价值函数和优化器。LATS在GPT-4上实现了HumanEval的SOTA pass@1 92.7%，超越了ReAct、Reflexion、CoT和ToT，但token成本更高。

**ToolTree**（2026）将双反馈MCTS与双向剪枝专门应用于工具使用规划，在保持规划质量的同时减少了探索开销。

### 2.3 层次化与分解式规划

**MapCoder**（Islam et al., 2024; ACL 2024）通过四个专门化智能体——检索、规划、编码和调试——复制了人类编程循环。规划智能体将问题分解为带有置信度分数的子步骤，引导动态遍历策略。MapCoder在HumanEval（93.9%）、MBPP（83.1%）和CodeContests（28.5%）上达到了SOTA。

**CodePlan**（2024; ICLR 2025）通过扩展代码形式的规划来释放推理潜力，其中计划以结构化伪代码而非自然语言表达，提升了规划质量和下游代码生成效果。

**GoalAct**（2025）引入全局规划与层次化执行，将智能体任务分解为高层技能（搜索、编码、写作），降低了网络和编程智能体的规划复杂度。

### 2.4 工作流式规划 vs. 自由形式规划

编程智能体中的一个关键架构区分是**自由形式规划**（如OpenHands CodeAct，赋予智能体在长周期规划中的广泛自由度）和**工作流约束规划**（如MoatlessTools、Agentless），后者预定义了动作空间和执行阶段。

- **Agentless**（Xia et al., 2024）认为智能体脚手架中的大部分感知复杂性可以用专门的提示和工作流设计来替代，在不需要智能体式自主性的情况下取得了有竞争力的结果。
- **MoatlessTools**通过预定义工作流约束模型的动作空间，有效缩短任务周期，使用Claude 3.5 Sonnet在SWE-bench上达到39%的解决率，每个问题仅需$0.14。

这一争论揭示了一个根本性张力：自由形式规划能够实现更具创造性的问题解决，但增加了上下文漂移和规划失败的风险；结构化工作流降低了方差，但可能错过新颖的解决路径。

## 3. LLM智能体的记忆系统

### 3.1 分类体系与认知基础

综述"Memory in the Age of AI Agents"（Liu et al., 2025）提供了基于认知科学的全面分类体系：

- **工作记忆**：智能体的活跃上下文窗口，类比于人类短期记忆
- **情景记忆**：过去交互和经验的记录（如Reflexion的语言反馈）
- **语义记忆**：关于世界的结构化知识（如知识图谱、代码仓库）
- **程序性记忆**：学习到的行为模式和技能

一个平行的进化框架（2026）提出了记忆发展的三个阶段：**存储**（轨迹保存）、**反思**（轨迹精炼）和**经验**（轨迹抽象）。

### 3.2 智能体式与自组织记忆

**A-MEM**（Xu et al., 2025）引入了受Zettelkasten方法启发的智能体记忆：记忆通过索引和链接被动态组织为互联的知识网络。四步流程——笔记构建、链接生成、记忆进化和检索——使记忆系统能够自我组织和进化。

**MemoryBank**（Zhong et al., 2024）将艾宾浩斯遗忘曲线应用于记忆管理，使用指数衰减模型来调节记忆强度。这实现了自然的优先级排序，频繁访问和最近的记忆被优先保留。

**层次化程序性记忆**（2025）通过贝叶斯选择和对比精炼从智能体轨迹中学习可复用的程序，支持跨任务的技能迁移。

### 3.3 面向生产的记忆系统

**Mem0**（2025）为生产环境智能体提供可扩展的记忆架构，相比OpenAI的记忆方案实现了26%的相对改进，p95延迟降低91%，token成本节省超过90%。增强版本使用基于图的记忆来捕获关系结构。

**MemGPT / Letta**（Packer et al., 2023）借鉴操作系统虚拟内存管理，引入层次化记忆系统，通过显式的页面调入/调出操作管理上下文。智能体通过函数调用在快速（上下文）和慢速（外部存储）记忆层级之间移动信息。截至2024年，MemGPT演变为Letta框架，推出了**Context Repositories**——面向编程智能体的基于git的持久化记忆。

### 3.4 编程智能体专用记忆

**Lore**（2026）将git提交消息重新利用为结构化知识协议，捕获论文所称的"决策影子"（Decision Shadow）——约束条件、被拒绝的替代方案、智能体指令和验证元数据，这些信息通常在提交后丢失。Lore仅需git基础设施，可通过CLI查询。

**Aider的RepoMap**提供代码仓库结构的简洁文本表示，突出显示与当前任务上下文最相关的部分。这相当于一种关于代码结构的压缩语义记忆。

**SWE-agent的Agent-Computer Interface (ACI)**（Yang et al., 2024; NeurIPS 2024）通过定制的文件导航、编辑和搜索接口增强智能体与代码的交互，隐式地将智能体的工作记忆围绕以代码为中心的操作进行结构化。

## 4. 上下文管理策略

### 4.1 上下文工程范式

"A Survey of Context Engineering for Large Language Models"（Mei et al., 2025）将上下文工程形式化为一门涵盖上下文检索、处理和管理的学科。**Agentic Context Engineering (ACE)**（2025）将上下文视为不断进化的策略手册，通过生成、反思和整理来积累和精炼策略，在智能体任务上实现了+10.6%的改进。

### 4.2 观察遮蔽 vs. LLM摘要

**JetBrains Research**研究（2025）和"Complexity Trap"论文（2025）为编程智能体提供了关键的实证洞察：

- **观察遮蔽**（隐藏除最近M个观察之外的所有内容）保留完整的动作和推理历史，同时大幅减少token数量
- **LLM摘要**将完整的交互轮次压缩为紧凑摘要，但存在信息丢失风险
- **关键发现**：简单的观察遮蔽在解决率上匹配或超越LLM摘要，同时成本降低约52%，因为观察token占据上下文的主导地位，而思维链推理得到更好的保留

使用Qwen3-Coder 480B模型时，观察遮蔽相比无管理的上下文在SWE-bench Verified上提升了2.6%的解决率，两种方法都将成本降低了50%以上。

### 4.3 结构化上下文管理

**Git Context Controller (GCC)**（Wang et al., 2025）是一个突破性框架，使用版本控制语义管理智能体上下文：

- **COMMIT**：在里程碑处持久化总结进展的检查点
- **BRANCH**：为替代推理路径创建隔离工作空间
- **MERGE**：将成功的分支整合回主轨迹
- **CONTEXT**：层次化检索相关历史上下文

GCC在SWE-bench Verified上达到了SOTA（使用强模型时成功率>80%），超越了26个现有系统，相比长上下文基线提高了>13%的任务解决率。这代表了结构化上下文管理是编程智能体性能关键差异化因素的最有力证据。

### 4.4 上下文折叠与主动管理

**Context-Folding**（2025）使智能体能够按程序分支进入子轨迹，并在完成后将其折叠，压缩中间步骤同时保留简洁的结果摘要。FoldGRPO强化学习框架训练智能体自主管理上下文，在活跃上下文缩小10倍的情况下达到与ReAct基线相当的性能。

**CORAL（认知资源自分配）**（2025）通过赋予智能体主动优化其上下文的能力来应对认知过载，防止注意力稀释导致的长周期规划退化。

**AgentFold**（2025）保持高度简洁的工作上下文（100轮交互后约7k token），支持扩展到500轮以上的网络智能体交互。

### 4.5 事务性上下文管理

**SagaLLM**（Chang et al., 2025; VLDB 2025）将Saga事务模式与持久化记忆集成用于多智能体规划：

- 显式保存目标、理由和依赖关系
- 模块化检查点和可补偿执行用于恢复
- 自动补偿和独立验证智能体

这种受数据库启发的方法解决了上下文丢失、不可靠的自我验证和不充分的跨智能体协调——这些是多智能体编程系统中的常见失败模式。

## 6. 差距分析与研究机会

### 6.1 差距：统一的规划-记忆架构

**现状**：规划和记忆/上下文管理在很大程度上被视为正交的关注点。GCC是一个值得注意的例外，但即使GCC的规划也是隐式的（通过BRANCH/MERGE），而非显式的层次化任务分解。

**机会**：设计规划系统和记忆系统协同设计的架构——规划决策指导记忆什么（和遗忘什么），记忆内容塑造规划策略。对于编程智能体，这意味着规划器能够同时推理代码仓库结构（语义记忆）、过去的调试尝试（情景记忆）和已知的修复模式（程序性记忆）。

### 6.2 差距：自适应上下文管理策略

**现状**：大多数上下文管理策略是静态的（固定遮蔽窗口、固定摘要触发器）。Complexity Trap论文表明简单方法表现出奇地好，但随着任务复杂度增加这一结论可能不成立。

**机会**：开发基于RL训练或元学习的上下文管理策略，根据任务难度、阶段（探索vs.利用）和智能体置信度自适应调整。Context-Folding的FoldGRPO是这个方向的一步，但聚焦于子任务结构而非自适应粒度。

### 6.3 差距：跨会话和跨任务的记忆迁移

**现状**：大多数编程智能体将每个问题视为独立的。Lore和Letta的Context Repositories是持久化记忆的早期尝试，但缺乏跨任务选择性迁移和遗忘的机制。

**机会**：构建能够跨会话积累仓库特定知识（架构模式、常见缺陷类型、团队约定）并选择性地将此知识应用于新任务的记忆系统——类似于经验丰富的开发者构建代码库心智模型的过程。

### 6.4 差距：规划+记忆的评估框架

**现状**：SWE-bench评估端到端的任务解决率，但不分解规划质量和记忆效果的各自贡献。目前没有专门衡量智能体在编程任务中长周期上下文保持能力的基准测试。

**机会**：开发诊断性基准测试，分别衡量：（a）规划深度和适应性，（b）压力下的上下文保持，（c）面向代码特定查询的记忆检索精度，（d）随任务周期增长的优雅退化。

### 6.5 差距：多智能体记忆协调

**现状**：SagaLLM解决了事务保证问题，但聚焦于一般规划。多智能体编程系统缺乏有原则的记忆共享协议——缺陷定位智能体应如何将相关上下文传递给补丁生成智能体？

**机会**：为多智能体编程工作流设计记忆共享协议，最小化冗余探索同时最大化专门化智能体之间的相关上下文传递。

## 7. 关键趋势与综合

1. **从提示工程到上下文工程**：该领域已明确超越了提示词的精心设计，转向设计完整的信息管理系统。对于编程智能体，这意味着将代码仓库上下文、执行轨迹和推理历史作为一等公民来管理。

2. **简单可以取胜**：Complexity Trap的结果——观察遮蔽优于LLM摘要——挑战了更复杂的记忆管理总是更好的假设。正确的基线至关重要。

3. **版本控制作为记忆隐喻**：GCC、Lore和Letta的Context Repositories都借鉴了git语义来实现智能体记忆。对于版本控制本身就是领域一部分的编程智能体来说，这是一种自然的契合。

4. **强化学习用于上下文管理**：Context-Folding的FoldGRPO和SWE-Gym的智能体训练表明，学习管理上下文（而不仅仅是学习编码）是一个可行且重要的训练目标。

5. **100万token上下文窗口是必要但不充分的**：Claude Code的成功证明了大上下文窗口的价值，但GCC在大上下文基础上的结构化管理取得了更好的结果。原始上下文容量是基础，而非解决方案。
