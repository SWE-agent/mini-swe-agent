# 研究提案：面向长周期编程智能体的自适应规划-记忆协同设计

**日期：** 2026-03-19
**基于：** 对规划与记忆/上下文工程约45篇论文的文献综述

---

## 1. 问题陈述

基于LLM的编程智能体在SWE-bench等基准测试上取得了令人瞩目的成果，但在需要跨多个步骤持续推理的长周期、多文件任务上仍存在系统性失败。我们的文献综述识别出一个关键差距：**规划系统和记忆/上下文管理系统是独立设计的**，导致次优的交互：

- 规划系统生成子目标时不考虑记忆系统能保留什么
- 记忆系统压缩或丢弃上下文时不考虑规划器需要哪些信息
- 两个系统都不根据任务阶段（探索vs.利用）或难度调整策略

迄今最成功的系统——GCC（Git Context Controller）——通过隐式耦合规划（BRANCH/MERGE）与记忆（COMMIT/CONTEXT）达到SOTA，但这种耦合是结构性的而非学习得到的。我们提议将这种耦合变为**显式的、自适应的、可学习的**。

---

## 2. 研究问题

**RQ1**：协同设计的规划-记忆架构能否在长周期编程任务上超越独立设计的规划和记忆系统？

**RQ2**：强化学习能否训练智能体联合优化规划决策和上下文管理策略？

**RQ3**：跨会话的情景记忆（积累的代码仓库知识）如何影响重复任务类型上的规划效率？

---

## 3. 提议方法：PlanMem

我们提出**PlanMem**，一种为编程智能体协同设计层次化规划与自适应上下文管理的架构。

### 3.1 架构概览

```
┌─────────────────────────────────────────────────────┐
│                    PlanMem 智能体                     │
│                                                      │
│  ┌──────────────┐    ┌──────────────────────────┐   │
│  │   规划器      │◄──►│   记忆控制器              │   │
│  │              │    │                          │   │
│  │ - 目标栈     │    │ - 工作记忆 (WM)          │   │
│  │ - 子任务     │    │ - 情景存储 (ES)          │   │
│  │   分解器     │    │ - 语义索引 (SI)          │   │
│  │ - 阶段      │    │ - 上下文预算分配          │   │
│  │   检测器     │    │ - 保留策略               │   │
│  └──────┬───────┘    └──────────┬───────────────┘   │
│         │                       │                    │
│         ▼                       ▼                    │
│  ┌──────────────────────────────────────────────┐   │
│  │          联合策略网络 (π)                      │   │
│  │   决策：做什么 AND 记住什么                     │   │
│  └──────────────────────────────────────────────┘   │
│                        │                             │
│                        ▼                             │
│  ┌──────────────────────────────────────────────┐   │
│  │           代码环境（沙箱）                      │   │
│  │   代码仓库 + Shell + 测试套件 + Git            │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

### 3.2 核心组件

#### A. 层次化目标感知规划器

规划器维护一个显式的目标栈，支持子任务分解：

1. **目标注册**：每个高层目标（如"修复issue #123"）与所需上下文（问题描述、相关文件、测试期望）一起注册
2. **子任务分解**：目标被分解为子任务（故障定位 → 理解代码 → 生成补丁 → 验证修复），每个子任务附带估计的上下文需求
3. **阶段检测**：规划器检测智能体当前所处阶段（探索、假设形成、实现、验证），并向记忆控制器发送信号以相应调整保留策略

#### B. 自适应记忆控制器

记忆控制器管理三个层级：

1. **工作记忆 (WM)**：活跃的上下文窗口，通过学习到的保留策略在每一步决定保留、压缩或驱逐什么
2. **情景存储 (ES)**：过去动作、观察和推理的结构化日志，按任务阶段和文件路径索引。使用选择性检索而非完整回放
3. **语义索引 (SI)**：代码仓库结构、代码语义和积累知识的持久化表示（类似RepoMap但动态更新）

#### C. 联合策略网络

PlanMem不使用分别用于动作选择和上下文管理的独立策略，而是使用一个**联合策略**，输出：
- **动作**：下一步调用什么工具（编辑文件、运行测试、搜索代码等）
- **记忆操作**：什么提交到情景存储、什么从工作记忆驱逐、什么从语义索引检索
- **规划信号**：是否进一步分解当前子任务、回溯、或声明子任务完成

### 3.3 训练方法

我们提出两阶段训练策略：

**阶段1：从专家轨迹进行行为克隆**
- 收集来自强基线（配备GCC的智能体、OpenHands）的成功轨迹
- 标注轨迹中的隐式规划决策和上下文管理操作
- 通过监督学习在标注轨迹上训练联合策略

**阶段2：带有过程奖励的强化学习**
- 使用SWE-Gym作为训练环境
- 定义过程奖励：
  - 规划质量：子任务完成率、回溯频率
  - 记忆效率：上下文利用率（相关token / 总token）
  - 总体：问题解决率 + 成本效率
- 使用GRPO的变体进行训练（类似Context-Folding的FoldGRPO）

### 3.4 跨会话记忆

PlanMem包含一个受Lore和GCC启发的**持久化记忆层**：

- 每个任务完成后，成功的策略和仓库特定知识被蒸馏到语义索引中
- 在新任务上，规划器查询语义索引获取相关的先前经验
- 受MemoryBank的艾宾浩斯曲线启发的遗忘机制防止陈旧知识的积累

## 4. 实验设计

### 4.1 基准测试

| 基准测试 | 目的 | 指标 |
|---------|------|------|
| SWE-bench Verified（500个实例） | 标准编程智能体评估 | 解决率，每实例成本 |
| SWE-EVO | 长周期软件演化 | 多步解决率 |
| SWE-bench Lite | 效率比较 | 固定预算下的解决率 |

### 4.2 基线方法

1. **ReAct**（无结构化规划或记忆）
2. **ReAct + 观察遮蔽**（简单上下文管理，根据Complexity Trap的强基线）
3. **GCC**（结构化上下文管理，当前SOTA）
4. **OpenHands CodeAct**（自由形式规划）
5. **MoatlessTools**（工作流约束规划）
6. **Context-Folding**（学习的上下文管理，无显式规划）

### 4.3 消融实验

- **PlanMem-NoPlanner**：仅记忆控制器（无层次化规划的上下文管理）
- **PlanMem-NoMemory**：仅规划器（无自适应上下文管理的层次化规划）
- **PlanMem-Static**：联合架构但使用固定（非学习的）策略
- **PlanMem-NoCrossSession**：无跨任务持久化记忆

### 4.4 诊断指标

除解决率外，我们还衡量：
- **上下文利用率**：与当前子任务相关的上下文token比例（通过回顾性分析衡量）
- **规划效率**：完成的子任务数 / 尝试的子任务总数
- **目标保持**：智能体的动作是否随时间保持与原始目标的一致性（衡量上下文漂移）
- **恢复率**：智能体通过回溯从失败子任务中成功恢复的频率

## 8. 与相关工作的定位

PlanMem与以下工作的差异化：
- **GCC**：GCC通过git语义提供规划和记忆之间的结构性耦合；PlanMem通过联合策略网络提供学习的、自适应的耦合
- **Context-Folding**：聚焦于子任务上下文管理，但缺乏显式的层次化规划和跨会话记忆
- **SagaLLM**：为多智能体系统提供事务保证；PlanMem聚焦于单智能体的规划-记忆协同优化
- **A-MEM**：自组织记忆，但没有任务感知的规划集成
- **MapCoder**：面向竞赛编程的多智能体规划；PlanMem面向具有持久化记忆的仓库级软件工程

## 9. 风险与缓解措施

| 风险 | 缓解措施 |
|------|---------|
| RL训练不稳定 | 从行为克隆开始；使用保守的GRPO更新 |
| Complexity Trap效应（简单基线胜出） | 将观察遮蔽作为强基线纳入；设计PlanMem能优雅退化到简单策略 |
| 跨会话记忆引入噪声 | 艾宾浩斯式遗忘 + 检索前验证 |
| 训练计算成本 | 使用SWE-bench Lite进行开发；扩展到Verified进行最终评估 |
| 架构对LLM骨干网络过于复杂 | 设计模块化组件，可以单独启用/禁用 |
