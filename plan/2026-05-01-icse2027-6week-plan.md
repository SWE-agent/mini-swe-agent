## Why ICSE (not NeurIPS/ICLR)

- ICLR 2027 deadline is **2026-10** (5 more months of risk; competitors keep publishing)
- NeurIPS 2027 deadline is **2027-05** (12 months too late)
- ICSE 2027 is **the next available top venue** that ranks A* and aligns with the empirical / SE-flavored framing of PlanMemAgent
- Reviewer profile: **SE researchers** — they value empirical rigor, threats-to-validity, repository-level evaluation, and human-meaningful framings; they are less obsessed with leaderboard SOTA than ML reviewers
- **ICSE has accepted 17 coding-agent papers across 2024–2026** (12 in 2026 alone); the venue is clearly hot for this topic
- The acceptable framing shifts: ICSE wants "**Context engineering for autonomous program repair at scale**" not "Planning+memory co-design for LLM agents"

## Six-week schedule

### Week 1 (2026-05-01 → 2026-05-08): Smoke + scope freeze
- **Mon–Tue:** Read threat-update report (≤ 1h dispatched). If a HIGH threat appears, pivot framing within 24h before committing experiments.
- **Tue–Wed:** Run smoke evaluation: `PlanMemAgent` vs `MemorySearchAgent` vs `DefaultAgent` on **20 SWE-bench Verified problems × Qwen3-Coder-30B local**. Goal: PlanMemAgent must beat MemorySearch by ≥ 5pp on 20 tasks before committing.
- **Wed:** Decision gate — if smoke fails, pivot to Lite (smaller benchmark, faster iteration) or fall back to ablating planner-only / memory-only as the contribution.
- **Thu–Fri:** Freeze paper outline (12 pages ICSE format). Section skeleton:
  - §1 Introduction (framing: context engineering for autonomous program repair)
  - §2 Background (mini-SWE-agent lineage, SWE-bench, prior planning+memory work)
  - §3 Approach (PlanMemAgent architecture diagram, phase machine, bidirectional loop)
  - §4 Implementation (mini-SWE-agent inheritance, configurations, costs)
  - §5 Empirical evaluation
  - §6 Threats to validity
  - §7 Related work
  - §8 Discussion + Future work (training, multi-agent)
- **Fri:** Finalize compute plan & API budget request.

**Deliverables:** smoke report; outline frozen; threat assessment done; compute approved.

### Week 2 (2026-05-09 → 2026-05-15): Headline experiments — main grid
- **Run Verified-500 grid** with:
  - 3 internal: DefaultAgent / MemorySearchAgent / PlanMemAgent
  - 2 external scaffolds: OpenHands+CodeAct, Agentless
  - 2 models: Qwen3-Coder-30B (local), Claude-Sonnet-4 (remote)
  - Total cells: 5 × 2 = 10 configurations × 500 tasks = 5000 task runs
  - Estimated wall-clock: ~3 days on 4 concurrent workers
- **In parallel:** start §3 Approach draft (architecture diagram, equations for beam search + IDF + MMR, phase machine state diagram)

**Deliverables:** headline tables for both models; §3 first draft.

### Week 3 (2026-05-16 → 2026-05-22): Ablation matrix + §4
- **Ablation matrix on Verified-500 × Sonnet-4:**
  - `{enable_planner ∈ {0,1}} × {enable_adaptive_memory ∈ {0,1}}` → 4 cells
  - Plus `enable_memory_to_planner ∈ {0,1}` sweep → +2 cells
  - Plus `enable_replanning ∈ {0,1}` → +2 cells
  - Total: 8 configurations × 500 tasks
- **Per-phase breakdown:** report resolve-rate per phase (how often does each phase fire, what's the success conditional on phase) — this is **distinctively novel** and reviewers haven't seen it
- **Cost / steps / failure-mode tables**
- **Draft §4 Implementation:** code-level documentation of phase detection heuristics, beam-search params per phase, AST graph construction
- **Draft §2 Background:** include the official SWE-agent → mini-SWE-agent supersession announcement

**Deliverables:** ablation tables done; §2, §4 first drafts.

### Week 4 (2026-05-23 → 2026-05-29): Case studies + §5/§6
- **Three case studies** (ICSE-style) — pick from SWE-bench Verified problems where PlanMemAgent succeeds but MemorySearch fails:
  - Case A: cross-file refactor (shows planner decomposition value)
  - Case B: failing-test-driven repair (shows backtrack + replan value)
  - Case C: long-context navigation (shows phase-aware retrieval value)
  - Each case: 1 page with trajectory excerpt, phase transitions, memory selections at critical steps
- **Draft §5 Evaluation:** structure with RQ1–RQ4
  - RQ1: Does PlanMem improve resolve rate over the inheritance baselines and external scaffolds? (headline table)
  - RQ2: Which components contribute most? (ablation)
  - RQ3: How do phase transitions correlate with task success? (per-phase breakdown)
  - RQ4: How sensitive is the method to phase-budget allocation? (sensitivity sweep)
- **Draft §6 Threats to validity:**
  - Internal: data leakage (SWE-bench Verified vs Qwen3-Coder training cutoff), randomness control
  - External: only Python repos, only SWE-bench
  - Construct: resolve@1 vs effort-adjusted metrics
  - Conclusion: comparison to LIVE-SWE-AGENT (sister system, larger budget)

**Deliverables:** §5, §6 drafts; case studies written.

### Week 5 (2026-05-30 → 2026-06-05): Related work + introduction + figures
- **Draft §7 Related work** following the `planmem_similarity_analysis.md` recommended organization:

  - Para 3: We close this gap on the mini-SWE-agent scaffold (the canonical successor to SWE-agent, 25 lineage citations)
  - Para 4: Contributions (4 bullets)
- **Figures:**
  - Fig 1: Architecture diagram (4 modules)
  - Fig 2: Phase state machine
  - Fig 3: Headline bar chart (resolve rate × scaffold × model)
  - Fig 4: Ablation matrix heatmap
  - Fig 5: Per-phase success heatmap

**Deliverables:** §1, §7 first drafts; all figures drafted.
