# PlanMem Non-Prompt-Engineering Fix Plan — P0 / P1 / P2

Status: design draft, awaiting approval. **Do not implement until reviewed.**

## 0. 已知约束（必须满足，否则诱发新 bug）

这些是从 v1 ~ v2 实验中学到的硬约束。每个 Fix 都必须不破坏它们。

| 约束 | 来源 | 表现 |
|---|---|---|
| **C-1 Toolcall pairing 不变** | toolcall API 规范 | `assistant.tool_calls[id=X]` 必须紧跟 `role:tool, tool_call_id=X`。任何 messages 修改不能破坏这个配对（v1 smoke #1-3 的 250 步空 patch 就是因为破坏了它） |
| **C-2 Submit protocol 不漂移** | v1 的 482/500 failed | 不能往 system prompt 注入"暗示替代提交方式"的内容。Qwen3-Coder 对 system 末尾的额外指令非常敏感，会被诱导走 `cat > patch.txt << EOF` 这条会被自我截断的死路。注入必须**短**、**指令化弱**、**不暗示编辑 patch 文件** |
| **C-3 Prompt cache 不抖动** | OpenAI/Anthropic 缓存策略 | 高频变化的字符串注入会使 prefix cache miss，cost & latency 大增。注入内容必须**慢变**（phase + active sub-task name，不是 step-count、不是 time-stamp） |
| **C-4 LLM 调用全部入账** | n_calls/cost 都在 agent 上 | 任何新加的 LLM 调用必须走 `self._accounted_query` 而非直接 `self.model.query` |
| **C-5 Config 全部 Pydantic、默认值齐全** | v0 的 dataclass-on-BaseModel bug | 新增 config 字段必须有 default，且类型与 Pydantic 兼容 |
| **C-6 新功能默认 OFF** | 防止 regression | 每个 Fix 加 `enable_*` flag，**默认 false**。只在 swebench_planmem.yaml 中显式 ON。便于 ablation 和回滚 |
| **C-7 全部 28 个现有测试不能破** | tests/agents/test_planmem.py | 每个修改后 `pytest tests/agents/test_planmem.py` 必须 28 passed |
| **C-8 不重复 v1 错误** | 反思 | 不要为了"看起来 PlanMem 更复杂"就堆功能。每个 Fix 加之前要能用一句话回答："这个 Fix 期望在 SWE-bench 上具体改善哪类失败" |

## 1. P0 — Fix A：Planning Header in System Prompt

**目标**：把 planner 的高层状态（current phase + active sub-task）以**极简、稳定**的形式注入 system prompt 末尾，让模型"知道自己在干什么"，但不破坏 submit protocol。

### 设计

- 在 `messages[0]["content"]` 末尾追加一个**marker 区块**（与 repo card 同样的模式，但内容完全不同）：

```
<!-- PLANMEM_HEADER:BEGIN -->
Phase: implementation
Current goal: Fix the bug in astropy/modeling/separable.py
<!-- PLANMEM_HEADER:END -->
```

- **最大长度**：≤ 200 字符。**禁止包含**：文件名列表、命令示例、"do/use/run" 这类指令式动词、submit 协议提示
- **更新条件**：only when `(phase, active_subtask.id)` 元组与上次注入不同。否则 messages 不动 → cache 不抖
- **idempotent**：strip 旧块再写新块；连续调用 N 次结果 = 调用 1 次

### 解决了什么、不解决什么

- ✅ 解决：模型不知道当前在 EXPLORATION 还是 IMPLEMENTATION（缺乏元认知）
- ✅ 解决：当 active sub-task 变化时，模型能看到新的子目标聚焦点
- ❌ 不解决：不能让模型"看到" memory graph 的 retrieval 结果（那个还是 dead）
- ❌ 不解决：不能让模型"看到" replan 后的新 sub-task 树（只能看到栈顶）

### 实施步骤

1. `planmem_agent.py` 加 `_apply_planning_header()` 方法（仿照 `_ensure_repo_background_card`）
2. `PlanMemConfig` 加：
   - `enable_planning_header: bool = False`
   - `planning_header_max_chars: int = 200`
3. `query()` 中：limit check 后、`_update_planner` 后调用
4. 加测试：
   - `test_planning_header_renders_phase_and_subtask`
   - `test_planning_header_idempotent_on_same_state`
   - `test_planning_header_disabled_no_change`
   - `test_planning_header_skips_when_unchanged` （cache 友好性）
   - `test_planning_header_strict_max_chars`
5. swebench_planmem.yaml 设 `enable_planning_header: true`

### 风险 & 缓解

| 风险 | 缓解 |
|---|---|
| 内容漂移导致 v1 那种 protocol 破坏 | 字符上限 200 + 内容白名单（只允许 phase 字符串 + sub-task description 截断） |
| Cache miss | "unchanged → skip update" 短路 |
| 配置错误导致空 prompt | 在 yaml 默认值 + 单元测试断言 |

### 预期 Δ（SWE-bench Verified）

+1 ~ +3 pp。理论上**最大**收益是让模型在卡住时自我意识到"还在 exploration、需要进 impl"。

## 2. P1a — Fix B：Per-Phase Sampling Routing

**目标**：根据 planner 的 phase 信号，动态调整 `temperature`（可选 `top_p`），不经过 prompt 走 API 参数路。

### 设计

| Phase | temperature | 理由 |
|---|---|---|
| EXPLORATION | 0.3 | 鼓励搜索多样性 |
| HYPOTHESIS | 0.1 | 推理需要较确定 |
| IMPLEMENTATION | 0.0 | 代码必须确定 |
| VERIFICATION | 0.0 | 测试命令也确定 |
| BACKTRACK | 0.4 | 高发散，尝试新角度 |

### 实施步骤

1. `PlanMemConfig` 加 `phase_sampling: dict[str, dict] = {...defaults above...}`
2. 加 `enable_phase_sampling: bool = False` flag
3. `PlanMemAgent.query()` 在调 model 前：
   ```python
   sampling_kwargs = {}
   if self.config.enable_phase_sampling and self._planning_signal is not None:
       sampling_kwargs = dict(
           self.config.phase_sampling.get(
               self._planning_signal.current_phase.value, {}
           )
       )
   response = self.model.query(self.messages, **sampling_kwargs)
   ```
4. 验证 `LitellmModel._query` 传 kwargs 给 litellm 是 OK 的（看代码确认 `**(self.config.model_kwargs | kwargs)` 这一行）
5. 加测试：
   - `test_phase_sampling_propagates_to_model`（mock model.query 检查 kwargs）
   - `test_phase_sampling_disabled_no_kwargs`
6. swebench_planmem.yaml 设 `enable_phase_sampling: true`

### 解决了什么、不解决什么

- ✅ 这才是 **planning → 非 prompt 通路 → 影响 LLM** 的真实例子
- ✅ Reproducibility 不变：固定 seed 时 deterministic
- ❌ 仅 sampling 没办法解决模型不知道 phase 是什么的问题（需要配合 P0）

### 风险 & 缓解

| 风险 | 缓解 |
|---|---|
| temperature > 0 引入随机性 | 在 model_kwargs 设固定 `seed` 让结果可复现 |
| litellm 某些 provider 不支持 seed | 文档化：vLLM/OpenAI 支持，Anthropic 部分支持 |
| EXPLORATION temp=0.3 可能让模型偏离最优 | 跑 50-instance subset 对照 v2 验证 Δ 后再决定 |

### 预期 Δ

+0.5 ~ +1.5 pp（独立贡献小，但与 P0 配合时显著）。

## 3. P1b — Fix D：Trajectory Rewind on Replan

**目标**：当 `replan_on_backtrack` 触发时，**真的把 `self.messages` 截短到失败 sub-task 起点之前**，让 agent 物理上"忘记"死路径，从干净状态重新出发。

### 设计

#### 关键不变量（必须证明）

**Rewind 必须在 message 边界上切**。如果在 `assistant.tool_calls` 之后但 `tool` 之前切，破坏 toolcall pairing → API 报错。

**安全切点定义**：在某个 `role: user|tool` 消息**之后**、下一条 `role: assistant` **之前**的位置。换句话说，cut point 一定是「上一次 observation 接收完毕、下一次 assistant 还没动作」的瞬间。

#### 数据结构

在 `PlannerState` 加：
```python
subtask_start_msg_idx: dict[int, int] = field(default_factory=dict)
# subtask.id → 该 sub-task 成为 active 时 messages 列表的长度
```

#### 触发流程

1. **Sub-task 入栈时记录 boundary**：在 `_adopt_as_children` 内，将每个新 subtask 记录其 birth 时刻的 `len(self.messages)`。但 planner 不直接持有 agent 引用，所以 agent 在调用 `_adopt_as_children` 后立即查 stack 头部并记录。
2. **Replan 时计算 cut**：planner.replan_on_backtrack 返回**失败 sub-task 的 birth msg_idx**。
3. **Agent 应用 cut**：在 `query()` 下一次执行前：
   - 找最近的「安全切点」`safe_cut <= birth_idx`（往前找到第一个 user/tool boundary）
   - `self.messages = self.messages[:safe_cut]`
   - `self.memory_graph = self.memory_graph[:safe_cut]`（保持一致）
   - 注入一条短的 user 消息："Previous approach failed; reconsidering."

#### Config

- `enable_trajectory_rewind: bool = False`
- `rewind_reset_message: str = "Previous approach failed. Try a different angle."`

### 解决了什么、不解决什么

- ✅ 这才是 **planning 真正改变 conversation 物理状态** 的体现
- ✅ 释放 context budget：长任务跑到 200 步后，rewind 能把无效路径全删掉，给后续动作腾空间
- ❌ 信息损失：失败路径中的 grep 结果等也被删了；模型可能重复探索
- ❌ 只在 replan 触发时生效；replan 触发率不到 100%

### 实施步骤

1. `types.py` 给 `PlanningSignal` 加 `rewind_to_msg_idx: int | None = None`
2. `planner.py`：
   - `PlannerState.subtask_start_msg_idx` 字段
   - `replan_on_backtrack` 返回值变成 `(bool, Optional[int])`，第二项是 safe rewind 点
   - 但 planner 不知道当前 msg 数 — 需要让 agent 在调用 replan 前传入或者在 build_signal 时填入
3. `planmem_agent.py`：
   - `_update_planner` 中：sub-task 入栈时记录 `birth_idx = len(self.messages)`
   - 拿到 rewind 信号后，在 `query()` 顶部应用 cut：
     ```python
     if self.config.enable_trajectory_rewind and self._pending_rewind is not None:
         cut = self._safe_cut_point(self._pending_rewind)
         self.messages = self.messages[:cut]
         self.memory_graph = self.memory_graph[:cut]
         self.add_messages({"role": "user", "content": self.config.rewind_reset_message})
         self._pending_rewind = None
     ```
   - 加 `_safe_cut_point(target_idx)`：从 target_idx 往前找到第一个 user/tool 边界
4. 测试：
   - `test_rewind_truncates_messages_at_safe_point`
   - `test_rewind_preserves_toolcall_pairing`（拼一个含 assistant.tool_calls 的假对话，断言 cut 后没有 orphan）
   - `test_rewind_disabled_no_truncation`
   - `test_rewind_resets_memory_graph_consistently`

### 风险 & 缓解

| 风险 | 缓解 |
|---|---|
| Cut point 算错 → API 报 400 | 严格的 `_safe_cut_point` + 单元测试覆盖所有 corner case |
| 频繁 rewind 让模型反复重做相同事情 | `replan_cooldown_steps` 已有；rewind 也走这个 cooldown |
| Memory graph 和 messages 长度不一致 | 共用 cut，两个一起截，附加单元测试断言长度一致 |
| Agent run loop 死循环（rewind → 同样 fail → 再 rewind） | replan 已有 cooldown，且失败 sub-task 进 failed_subtasks 不会再被选 |

### 预期 Δ

+2 ~ +5 pp（在 step_limit 触发率较高、context 紧张的 instance 上更明显）。

## 4. P2 — 三选一（讨论后定）

### 选项 P2-α：Active Summarization

把 memory_graph 中老的 user/tool message 用一次 LLM 调用压成 1 个 summary node。

**好处**：长任务延长有效 horizon、PlanMem 真有 memory 算法在 work。
**坏处**：toolcall pairing 棘手（删多条 tool message 需要同时删对应 assistant.tool_calls 或保留 pair 整体）。**实施难度最大**。

### 选项 P2-β：Sub-agent Dispatch

planner 判定某个 sub-task 需要"全新视角"时，spawn 一个 fresh `DefaultAgent` 处理那个子任务，返回 summary。

**好处**：架构层面与 prompt engineering 完全无关；多层规划真实落地。
**坏处**：状态管理复杂、cost 翻倍、回归测试难写。

### 选项 P2-γ：Output Validation + Retry

LLM 输出 patch 后，agent 在 docker 内 `git apply --check`，失败则直接重 query 不入 messages。

**好处**：实现简单、直接攻克 patch 失败问题（v1 的 482/500 主要就是 patch 不能 apply）。
**坏处**：与 PlanMem 主线无关，只是工程改进。

**我的建议**：P2 选 γ（output validation）作为最快胜利，然后再考虑 α 或 β 作为论文级别 contribution。

## 7. TL;DR

**P0（planning header）**：低风险、直接补 dead-mechanism 的 prompt 通路，但**字符 ≤ 200 + 无指令性内容**避免 v1 protocol 破坏。

**P1a（sampling routing）**：通过 API 参数路而非 prompt 影响 LLM，与 P0 正交。

**P1b（trajectory rewind）**：planning 真正改变 conversation 物理状态——非 prompt 通路、PlanMem 学术意义上的核心机制。**最大风险是 toolcall pairing 切错位置**，必须严格测试。

**P2**：α 难、β 大、γ 简单实用 — 建议先 γ。
